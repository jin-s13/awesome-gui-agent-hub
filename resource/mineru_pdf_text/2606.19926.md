## Abstract

MLLM-based mobile GUI agents have made substantial progress on short-horizon tasks, yet remain unreliable on long-horizon tasks that require retaining intermediate facts across many steps and app transitions. We attribute this limitation to ReAct-style prompting, which passively accumulates per-step records, leading to prompt explosion and dilution of critical cross-app facts. To address this, (i) we introduce MemGUI-Agent, an end-to-end long-horizon mobile GUI agent with proactive context management. MemGUI-Agent is built on Context-as-Action (ConAct), which casts context management as first-class actions emitted by the same policy that selects UI actions. Instead of passively appending history, <sup>ConAct</sup> maintains three structured context fields: folded action history, folded UI state, and recent step record, preserving critical UI facts while keeping context compact. To make proactive context management learnable across model scales, (ii) we construct MemGUI-3K, a 2,956-trajectory dataset with full <sup>ConAct</sup> annotations for supervised training and ofline analysis. (iii) Training an 8B model on MemGUI-3K produces MemGUI-8B-SFT, an 8B MemGUI-Agent that achieves the best open-data 8B performance on MemGUI-Bench and generalizes to the out-of-distribution MobileWorld benchmark. Code, data, and trained models will be released at https://memgui-agent.github.io/.

## 1 Introduction

Recent MLLM-based mobile GUI agents can understand screenshots, reason over user goals, and control devices [6, 8, 11, 20, 30]. However, they remain unreliable in long-horizon settings that require retaining intermediate facts, tracking multi-step progress, and carrying information across app boundaries [9, 12]. As task horizons grow, even the strongest 8B end-to-end model, GUI-Owl-1.5-8B [20], drops from 71.6% on AndroidWorld [16] (avg. 8.4 steps) to 38.2% on MobileWorld [9] (avg. 27.8 steps), and further to 11.7% on MemGUI-Bench [12] (avg. 36.2 steps).

We attribute this degradation to context management. Agentic frameworks couple a backbone MLLM with planning, memory, and grounding modules [1, 16, 18, 19], but rely on complex pipelines and often proprietary backbones. End-to-end models are simpler and more deployable [6, 20, 30], yet manage context through passive strategies (e.g., Action-Thought, Multi-turn, Rule-based, No History; see Figure 2 and Table 2) that append, truncate, or discard records mechanically rather than according to task needs. Consequently, prompts grow with task horizon (prompt explosion), while critical cross-app facts such as prices, identifiers, or copied text may be diluted, paraphrased, truncated, or forgotten (information loss) [12]. We review related mobile GUI agents and context-management methods in Appendix A.

This motivates a policy-level mechanism that decides what to compress, what to remember, and what to keep available while acting

Can a single end-to-end mobile GUI agent jointly choose UI actions and manage its working context, so that long-horizon interaction stays compact while critical UI facts persist across steps and app transitions?

We answer this question with MemGUI-Agent, an end-to-end long-horizon mobile GUI agent with proactive context management. MemGUI-Agent is built on Context-as-Action (ConAct), which casts context managemen as first-class actions emitted alongside UI actions. Instead of passively appending history, <sup>ConAct</sup> maintains three structured context fields: Folded Action History H<sub>t</sub> for compressed trajectory summaries, Folded UI State M<sub>t</sub> for persistent UI-derived facts, and Recent Step Record L<sub>t</sub> for recent interaction details. All three are predicted in a single forward pass, keeping context curation inside the agent policy.

As shown in Figure 1(a), <sup>ConAct</sup> keeps prompt growth flatter than ReAct-style prompting. Figure 1(b) shows that this lower context cost translates into stronger long-horizon task success: applied zero-shot to Qwen3-VL-235B-Thinking, MemGUI-Agent reaches 62.5% Pass@3 on MemGUI-Bench, surpassing agentic workflows built on Gemini-2.5-Pro. To make proactive context management learnable across model scales, we construct MemGUI-3K, a 2,956-trajectory dataset with full <sup>ConAct</sup> annotations. Training an 8B model on MemGUI-3K produces MemGUI-8B-SFT, which achieves the best open-data 8B performance on MemGUI-Bench and generalizes to the out-of-distribution MobileWorld benchmark.

## 2 MemGUI-Agent: End-to-End Mobile GUI Agent with ConAct

<sup>ConAct</sup> makes context management a first-class part of the action policy in an end-to-end mobile GUI agent. Instead of treating context as a passive log outside the model’s control, the agent jointly decides what UI or memory action to execute, what history to fold, and how to describe the current interaction. This turns context maintenance into policy-level behavior. Figure 3 illustrates the single-step execution flow.

Problem formulation. We formulate mobile GUI automation as a sequential decision problem with a structured working context. Given task goal G and screenshot $I _ { t }$ , the agent observes

$$
\mathcal {S} _ {t} = (G, H _ {t}, M _ {t}, L _ {t}),\tag{1}
$$

where $H _ { t } , \ M _ { t } ,$ and $L _ { t }$ denote Folded Action History, Folded UI State, and Recent Step Record, storing compressed trajectory summaries, persistent UI-derived facts, and latest-step details. At each step, the MLLM policy emits a joint <sup>ConAct</sup> decision:

$$
\begin{array}{r} y _ {t} = (\tau_ {t}, \phi_ {t}, a _ {t}, o _ {t}, \iota_ {t}) \\ \sim \pi_ {\theta} (\cdot | I _ {t}, \mathcal {S} _ {t}), \end{array}\tag{2}
$$

where $\tau _ { t }$ is reasoning, $\phi _ { t }$ is a folding directive, $a _ { t }$ is a UI or memory action, $O _ { t }$ is the UI observation, and $\boldsymbol { l } _ { t }$ is the action intent. Unlike ReAct-style prompting, which appends a growing task-progress string, <sup>ConAct</sup> partitions context into three fields and updates them through model-emitted context actions.

Step output protocol. At each step, the model emits a 5-part structured output. We build on the Qwen3- VL function-calling prompt format [2], which asks for a concise thought and a JSON tool call enclosed by <tool\_call> tags. <sup>ConAct</sup> adds context-management fields for folding and self-description, and extends the executable tool-call action space with memory operations. The fields are colored by their roles, consistent with Figure 3:

The <tool\_call> field executes exactly one action per step from the extended action space

$$
\begin{array}{c} \mathcal {A} = \mathcal {A} _ {u i} \cup \mathcal {A} _ {m e m}, \\ \mathcal {A} _ {m e m} = \{\mathrm{add}, \mathrm{update}, \mathrm{delete} \}. \end{array}\tag{3}
$$

Here, $\mathcal { A } _ { u i }$ contains UI actions such as clicking, typing, swiping, waiting, and terminating. We use add, update, and delete as shorthand for memory\_add, memory\_update, and memory\_delete, which update $M _ { t }$ . Other context fields do not act on the environment; they specify how the working context is updated for future steps.

♣ History folding (over $H _ { t } )$ . From step 2 onward, the agent emits a mandatory <folding> block. Formally, the folding directive is $\phi _ { t } = ( [ s _ { t } , t ] , z _ { t } )$ , where $[ s _ { t } , t ]$ is the history span to compress and $z _ { t }$ is the generated summary. Step-level distillation corresponds to $s _ { t } = t$ and keeps the latest step as a compact record, while span-level abstraction uses $s _ { t } < t$ to summarize a completed sub-task into one reusable record. The folded history is updated as

$$
H _ {t + 1} = \operatorname{Fold} (H _ {t}, \phi_ {t}),\tag{4}
$$

which replaces uncontrolled <conclusion> accumulation with model-controlled folding, pushing context growth from linear accumulation toward sub-linear growth.

♠ UI memory actions (over M<sub>t</sub>). Each memory item is a structured triple $m = ( { \mathrm { i d } } , d , c )$ , where id is a unique identifier, d is a short description, and c is the complete content to preserve. Let $\mathcal { A } _ { m e m } = \{ a ^ { + } , a ^ { \circ } , a ^ { - } \}$ denote memory\_add, memory\_update, and memory\_delete. A memory action induces

$$
M _ {t + 1} = \left\{ \begin{array}{l l} \text {Add} (M _ {t}, m), & a _ {t} = a ^ {+}, \\ \text {Update} (M _ {t}, \text {id}, c), & a _ {t} = a ^ {\circ}, \\ \text {Delete} (M _ {t}, \text {id}), & a _ {t} = a ^ {-}, \\ M _ {t}, & a _ {t} \in \mathcal {A} _ {u i}. \end{array} \right.\tag{5}
$$

Memory writes store complete task-relevant information rather than references or lossy summaries, e.g., a full copied price or phone number instead of “the value seen earlier.” This is crucial when prices, codes, contact information, or copied text must survive screen changes, long delays, and app transitions.

♢ Self-describing step output (fills L<sub>t</sub>). The two self-describing fields make each step reusable by future context actions. <ui\_observation> provides a grounded screen description with exact visible text, numbers, names, and task-relevant UI facts. <action\_intent> states what the current tool call is intended to accomplish. Together with the executed action and tool result, they form

$$
L _ {t + 1} = (o _ {t}, \iota_ {t}, a _ {t}, r _ {t}).\tag{6}
$$

This record supplies grounded content for memory writes into $M _ { t }$ and future folding into $H _ { t }$ . Without these fields, folding and memory actions must infer step meaning only from raw <thinking> and <tool\_call> text, which we find insuficient when context actions are layered onto the ReAct 3-part protocol (§4.3).

Step execution loop. At step $t ,$ the MLLM $\pi _ { \theta }$ takes $( I _ { t } , S _ { t } )$ and emits $y _ { t } ~ = ~ ( \tau _ { t } , \phi _ { t } , a _ { t } , o _ { t } , \iota _ { t } )$ . The environment/tool result is

$$
(r _ {t}, I _ {t + 1}) = \left\{ \begin{array}{l l} \mathrm{Env} (I _ {t}, a _ {t}), & a _ {t} \in \mathcal {A} _ {u i}, \\ (\mathrm{ok}, I _ {t}), & a _ {t} \in \mathcal {A} _ {m e m}, \end{array} \right.\tag{7}
$$

where UI actions change the screen and memory actions update only the structured context. Combining Eq. 4, Eq. 5, and Eq. 6, the complete state transition is

$$
\begin{array}{c} \mathcal {S} _ {t + 1} = \mathcal {T} (\mathcal {S} _ {t}, y _ {t}, r _ {t}) \\ = (G, H _ {t + 1}, M _ {t + 1}, L _ {t + 1}). \end{array}\tag{8}
$$

The next state $\boldsymbol { S } _ { t + 1 }$ is fed into step t+1. Because folding, memory, and action intent are predicted by the same multimodal policy in one forward pass, compression and memorization inherit the agent’s task-leve reasoning rather than being delegated to a separate summarizer, retriever, or memory agent.

<sup>ConAct</sup> provides an end-to-end interface for folding history, writing UI memory, and describing recent steps, but using it well still requires policy decisions. We therefore ask whether the protocol alone is suficient across model scales. Table 1 applies the same 5-part <sup>ConAct</sup> protocol to Qwen3-VL backbones of diferent sizes on MemGUI-Bench-40. Gains emerge only on the strongest backbone we test, Qwen3-VL-235B-Thinking; on smaller open models and Qwen3-VL-235B-Instruct, the added protocol often reduces performance. This suggests that proactive context management is not merely a prompt-format change: models must learn when to fold history, when to write UI facts into memory, and how to produce reusable step descriptions. We therefore construct MemGUI-3K, a supervised dataset with full <sup>ConAct</sup> annotations, to support training and analysis across model scales.

## 3.2 Construction Pipeline

![](images/ec7e8c95531f32328a9241ec3856b1d030ce375632e1d2bcc4358ad522663caa.jpg)  
Figure 4 MemGUI-3K data collection and filtering pipeline. Expanded tasks are rolled out by a Qwen3-VL-235B-Thinking teacher, evaluated by MemGUI-Eval, filtered at trajectory and step levels, and converted into SFT samples.

Expanding seed tasks. MemGUI-3K is built from the 128 seed tasks of MemGUI-Bench [12]. We expand them with three complementary strategies: entity substitution, which preserves task structure while replacing task-specific entities; memory-operation augmentation, which increases coverage of memory\_update and memory\_- delete; and task simplification, which decomposes complex multi-app tasks into shorter single-objective variants. This produces a 7,303-task pool, from which 5,293 tasks enter rollout. Detailed expansion statistics are provided in Appendix C.1.

Collecting teacher rollouts. Each task is executed once in the snapshot-based Android environment of MemGUI-Bench using Qwen3-VL-235B-Thinking as the teacher with the full 5-part <sup>ConAct</sup> protocol. Rollouts terminate when the agent emits terminate or reaches the task-specific budget 2.5g + 1, where g is the task’s golden-step count. We evaluate trajectories with the MemGUI-Eval progressive scrutiny pipeline [12] and additionally annotate every step as reasonable or unreasonable. This filtering removes recovery loops, repeated errors, and counterproductive actions from SFT targets while preserving successful context-management behavior. Figure 4 visualizes the workflow; details and an annotated example are in Appendices C.2–C.4.

Filtering into training samples. After trajectory filtering and a 90/10 split, MemGUI-3K contains 2,956 successful trajectories across 26 apps, with verified zero overlap against the 128 MemGUI-Bench evaluation tasks. Extracting reasonable steps yields 64,430 SFT samples, including 57,951 train and 6,479 test samples. Each sample consists of the shared system prompt, the user message with structured context state and screenshot, and the gold 5-part <sup>ConAct</sup> response. We train MemGUI-8B-SFT from Qwen3-VL-8B-Instruct using LoRA SFT with ms-swift [28]; dataset format and training details are provided in Appendix C.6 and Appendix D.1.

## 4 Experiments

Evaluation setup. We evaluate on MemGUI-Bench [12], MobileWorld GUI-Only [9], and MemGUI-Bench-40 for ablations and failure analysis. We report Pass@k (k∈{1, 3}), IRR, and MTPR on MemGUI-Bench, and single-attempt success rate on MobileWorld. Baselines are listed in Tables 2 and 3; our two reported agents are MemGUI-Agent-235B, a zero-shot <sup>ConAct</sup> agent using Qwen3-VL-235B-Thinking, and MemGUI-8B-SFT, an SFT Qwen3-VL-8B-Instruct agent trained on MemGUI-3K. Unless otherwise stated, ablations, case studies, and failure analysis use zero-shot MemGUI-Agent-235B. Benchmark details and prompts are in Appendices B and H.

## 4.1 Main Results

Zero-shot <sup>ConAct</sup> sets a new SOTA on MemGUI-Bench. On MemGUI-Bench (Table 2), MemGUI-Agent-235B reaches 37.5% Pass@1 and 62.5% Pass@3, surpassing the best agentic framework, M3A on Gemini-2.5-Pro (32.8% / 47.7%), and improving over the same backbone by +13.3 Pass@1 and +16.8 IRR. On MobileWorld (Table 3), MemGUI-Agent-235B achieves 29.1% SR, +14.6 over the Qwen3-VL-235B-Thinking baseline, showing that zero-shot <sup>ConAct</sup> also transfers to a diferent environment and app set. As Figure 1(a) shows, these gains are not explained by larger prompts alone: <sup>ConAct</sup> keeps prompt growth substantially flatter than the ReAct baseline over 150 steps.

MemGUI-3K enables an 8B agent and transfers beyond its source benchmark. MemGUI-8B-SFT reaches 23.4% Pass@1 on MemGUI-Bench versus 9.4% for the Qwen3-VL-8B-Instruct baseline, with gains concentrated on harder tasks (Medium +19.0, Hard +18.5 vs. Easy +6.2). This dificulty trend indicates that MemGUI-3K does not merely teach a new output format; it improves behavior in the regime where long-horizon context drift is most severe. On MobileWorld, where the environment, apps, and evaluation protocol all difer, MemGUI-8B-SFT still reaches 17.9% SR (+8.5 over backbone) and ranks among the best open-data 8B models. The learned context-management skills therefore transfer beyond the source benchmark rather than only fitting MemGUI-Bench.

## 5 Conclusion

We presented MemGUI-Agent, an end-to-end long-horizon mobile GUI agent that manages context inside the action policy through <sup>ConAct</sup>, unifying history folding, UI memory actions, and self-describing step outputs. Zero-shot <sup>ConAct</sup> sets a new state of the art on MemGUI-Bench with Qwen3-VL-235B-Thinking. We further construct MemGUI-3K, enabling MemGUI-8B-SFT to achieve the best open-data 8B performance and generalize to MobileWorld.

## Limitations

Our experiments focus on Android-style mobile GUI environments. Extending <sup>ConAct</sup> to iOS, desktop, and web interfaces remains future work.

## A Related Work

Mobile GUI agents. Mobile GUI agents built on multimodal models generally follow two paradigms. Agentic frameworks combine a backbone MLLM with planning, memory, and grounding modules [1, 16, 18, 19], often around proprietary backbones such as Gemini-2.5-Pro [5]; they can perform strongly but are complex and dificult to reproduce. End-to-end models [6, 8, 20, 30] directly map screenshots to actions, ofering simplicity and deployability, but typically rely on passive context mechanisms: Action-Thought traces [2, 7, 20, 25], multi-turn prompts [15], rule-based aggregation [29], or no-history prompting [8]. These mechanisms either let prompts grow with task horizon or lose UI-derived facts needed across screens and apps [9, 12]. MemGUI-Agent brings proactive context management into the end-to-end paradigm itself.

Context management in LLM agents. Long-context LLM agents manage histories through external memory or prompt-internal curation. External memory maintains a separate store, such as retrieval-augmented databases [4, 10, 21], hierarchical memory [26], or experiential replay [27]. Prompt-internal curation edits the in-task context through truncation, summarization, or model-controlled compression, as in recent web-agent methods such as MEM1 [31], MemAgent [24], and AgentFold [23]. Unlike text-only web settings, mobile GUI control must preserve UI-derived facts, such as prices, codes, and copied text, verbatim across screen and app transitions. <sup>ConAct</sup> addresses this by treating context curation as joint action over structured history, UI state, and recent-step fields within the same end-to-end policy.

## B Benchmark and Online Evaluation Protocol

Benchmark details. MemGUI-Bench [12] contains 128 tasks across 26 Android apps, of which 89.8% are explicitly memory-intensive. Tasks are partitioned by golden-step count into Easy (1–20, 48 tasks), Medium (21–40, 42 tasks), and Hard (41+, 38 tasks); 78.1% are cross-app, with an average length of 36.2 steps. The 40-task subset MemGUI-Bench-40 is reserved for ablations. MobileWorld [9] GUI-Only contains 117 tasks in a containerized environment with disjoint apps (Mattermost, Mastodon, Mall4Uni), averaging 27.8 steps per task with 62.2% cross-app.

Compared agents. On MemGUI-Bench, we compare against agentic frameworks powered by Gemini-2.5-Pro: Agent-S2 [1], M3A/T3A [16], Mobile-Agent-E [19], Mobile-Agent-V2 [18], SeeAct [29], and AppAgent [25]; and end-to-end models including Qwen3-VL 8B/32B/235B variants [2], GUI-Owl-1.5 8B/32B [20], UI-TARS-1.5-7B [15], and UI-Venus-7B [7]. On MobileWorld, we additionally include ScaleCUA-7B [13], OpenMobile 7B/8B [3], and ClawGUI-2B [17].

MemGUI-Bench evaluation. For MemGUI-Bench and MemGUI-Bench-40, each task is evaluated with up to 3 independent attempts. For a task with golden-step count g, the maximum interaction budget is

$$
B _ {\mathrm{MemGUI}} (g) = 2. 5 g + 1,
$$

rounded to an integer in implementation. This adaptive budget gives longer tasks proportionally more room while preventing unlimited exploration. We report Pass@k for k∈{1, 3}, where a task is counted as successful if any of the first k attempts is judged successful by MemGUI-Eval.

MobileWorld evaluation. For MobileWorld GUI-Only, we follow the benchmark protocol and run a single attempt per task with a 50-step limit. We report Pass@1 success rate (SR), where task success is determined by the MobileWorld backend verifier.

Memory-oriented metrics. Besides success rate, MemGUI-Bench reports memory-oriented metrics. IRR (Information Retention Rate) measures whether task-critical information units are retained and correctly used. MTPR (Memory-Task Proficiency Ratio) measures success on explicitly memory-intensive tasks. These metrics are computed by the MemGUI-Eval pipeline [12].

## C.1 Task Expansion

We start from the 128 seed tasks of MemGUI-Bench [12], which are long-horizon mobile GUI tasks with strong cross-step dependencies. We expand them through three complementary strategies.

• Entity substitution preserves the original task pipeline but replaces task-specific entities, such as product names, search queries, contacts, notes, locations, or app-specific targets. This yields 456 expanded tasks.

• Memory-operation augmentation explicitly constructs tasks requiring not only memory\_add, but also memory\_update and memory\_delete. This yields 289 memory-operation-focused tasks.

• Task simplification decomposes complex expanded tasks into shorter, single-objective variants while preserving the need for structured context management. This yields 6,558 simplified tasks.

The three strategies produce a combined pool of 7,303 tasks, of which 5,293 are selected for rollout. The final dataset contains three source types: simplified tasks, entity-substitution tasks, and memory-operation augmentation tasks.

## C.2 Teacher Rollout and Trajectory Filtering

All rollouts are collected with Qwen3-VL-235B-Thinking using the full 5-part <sup>ConAct</sup> protocol described in §2. Each generated task is executed once in the snapshot-based Android environment of MemGUI-Bench. The environment is reset before each task and supports UI actions, such as click, swipe, type, wait, and system\_button, as well as memory actions, i.e., memory\_add, memory\_update, and memory\_delete. For a generated task with golden-step count g, the teacher rollout budget is also set to 2.5g + 1 steps, following the MemGUI-Bench evaluation protocol. Each rollout stops when the agent emits terminate or reaches this task-specific budget.

We first retain only successful trajectories according to the final decision of MemGUI-Eval [12]. MemGUI-Eval follows a progressive scrutiny pipeline: it first screens task completion from the goal, action logs, and final screenshots; then performs semantic analysis with generated step descriptions and memory-related checks; and finally verifies critical visual information against requested screenshots. Among 5,293 rollout trajectories, 2,959 receive a positive final decision. We then remove one abnormal 321-step outlier from an early collection run and two successful trajectories involving low-frequency apps with at most 8 occurrences. This produces the final 2,956 MemGUI-3K trajectories.

The final trajectories are split 90%/10% at the trajectory level with seed 42. We verify zero overlap against the 128-task MemGUI-Bench evaluation set by checking both task identifiers and task descriptions. Since MemGUI-Bench-40 is a subset of MemGUI-Bench, it is also disjoint from MemGUI-3K.

## C.6 Dataset Format

MemGUI-3K is released in two complementary formats. The trajectory-level format preserves complete rollouts, evaluation metadata, IRR statistics, token usage, screenshots, and step-level reasonableness annotations, supporting ofline analysis of memory operations, folding granularity, token cost, and recovery patterns. The step-level format converts all reasonable steps into ms-swift-compatible SFT samples. Each sample contains the shared system prompt, the user message with structured context state and current screenshot, and the gold 5-part <sup>ConAct</sup> response. Only steps labeled as reasonable are included in the SFT split.

The SFT data follows the standard chat-style multimodal format. A schematic example is shown below:

```json
{
```

## D Training Setup and Dynamics

## D.1 Training Setup

MemGUI-8B-SFT is initialized from Qwen3-VL-8B-Instruct and fine-tuned on the 57,951 reasonable-step training samples of MemGUI-3K using LoRA SFT with ms-swift [28]. Each sample contains the shared system prompt, the user message consisting of the structured context state and current screenshot, and the gold 5-part <sup>ConAct</sup> response. Training is conducted on eight 80GB GPUs and takes 535 minutes.

We use bfloat16 precision and train for one epoch with AdamW fused optimizer, cosine learning-rate scheduling, and a warmup ratio of 0.05. The per-device training batch size is 2 with gradient accumulation of 8, giving an efective batch size of 128 across 8 GPUs. We set the learning rate to $1 \times 1 0 ^ { - 4 }$ , weight decay to 0.1, maximum gradient norm to 1.0, and maximum sequence length to 32,768. LoRA uses rank 8, alpha 32, dropout 0.05, and is applied to all linear language-model projection modules. We use seed 42, four dataloader workers, and report training logs to Weights & Biases.

This section details the ofline skill evaluation protocol used in Table 4. Action matching. The evaluator parses JSON from the <tool\_call> block and follows a ClawGUI-style two-level protocol [17]. Type accuracy checks whether the predicted action type matches the gold action type. Match accuracy additionally checks action arguments. For click and long\_press, the predicted coordinate must fall within 14% of the $1 0 0 0 \times 1 0 0 0$ screen diagonal from the gold coordinate. For swipe, both start and end coordinates must fall within 1.5 times this threshold. For type and answer, the text must match exactly after stripping whitespace or achieve token-level $\mathrm { F 1 > 0 . 5 }$ . For system\_button and terminate, the button/status argument must match exactly; wait succeeds whenever the action type matches.

Memory-action metrics. For memory actions, type accuracy evaluates whether the predicted operation is memory\_add, memory\_update, or memory\_delete when the gold output requires the same operation. For memory\_delete, match success requires exact memory\_id match. For memory\_add and memory\_update, match success requires either description token F1 > 0.5 or content token F1 > 0.5. The description criterion is used because numeric contents such as prices may have low token overlap even when the predicted memory intent is correct. Trigger precision, recall, and F1 treat any memory action as a binary event over all steps and evaluate whether the model invokes memory at the same steps as the gold response.

Folding metrics. The evaluator parses JSON from the <folding> block. Presence accuracy checks whether folding is present or absent consistently with gold. Range accuracy checks whether both predicted range endpoints are within ±2 steps of the gold endpoints. A fold is considered span-level, or deep, if its range covers more than one step; otherwise it is shallow. We report predicted deep-fold ratio and separate range accuracy for shallow and deep folds, because a model can obtain a high overall range score by emitting mostly shallow folds.

Format compliance. Format compliance checks whether the output contains matched opening and closing tags for <thinking>, <folding>, <tool\_call>, <ui\_observation>, and <action\_intent>. The <folding> tag is treated as optional for step 1, matching the <sup>ConAct</sup> protocol. The evaluator also computes a trajectorylevel full-match rate, defined as the fraction of trajectories for which every evaluated step achieves action match success; we use this only as a diagnostic because the ofline setting fixes gold context at each step rather than rolling out model-induced states.

## G Failure Taxonomy and Annotation

Figure 7 categorizes failed attempts from the zero-shot Qwen3-VL-235B-Thinking ablation setting on MemGUI-Bench-40 into five dominant failure types. The annotation protocol follows the MemGUI-Bench failure taxonomy [12]. Each failed attempt is assigned to one primary category according to its main cause of failure.

Process hallucination. The agent loses track of the task objective or procedural workflow during execution, leading to goal drift, premature termination, irrelevant actions, skipped required steps, or a false belief that an intermediate operation has already been completed.

Output hallucination. The agent has observed task-relevant UI information, and often follows the correct workflow, but later stores, recalls, writes, calculates, or reports incomplete or incorrect facts. It includes selective retention failures over multi-item information and final-output transcription errors.

Knowledge deficiency. The agent lacks fundamental app, domain, or operational knowledge required for task completion, independent of whether it retained the needed information. Typical cases include misidentifying an app, misunderstanding a common UI convention, or not knowing the skill needed to continue.

Intent misunderstanding. The agent misinterprets the user instruction or task condition, then executes an action sequence for the wrong objective. The agent may correctly observe and retain facts but still fail because it optimizes for the wrong sub-goal, comparison criterion, destination, or target information.

Other. This category includes failures that do not fit the above classes, such as action-space limitations, environment instability, parsing errors, ineficient execution, or rare system-level failures.

Full <sup>ConAct</sup> mainly reduces process hallucination and output hallucination, while knowledge deficiency and intent misunderstanding remain comparatively stable. This supports the interpretation that the primary gains come from improved context management rather than unrelated model capabilities.
